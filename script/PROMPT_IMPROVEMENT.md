# VLMプロンプト改善 - 2026年5月18日

## 改善内容

スケッチ認識の精度向上のため、VLMのシステムプロンプトを改善しました。

### 主要な改善点

#### 1. **人間のスケッチに対する許容ルール (Tolerance) を追加**
```
従来: ピクセルレベルで厳密に判定
改善: 幾何学的不正確さ（線の途切れ、比率の狂い、歪み）を許容
```

#### 2. **セマンティック意図（Semantic Intent）を重視**
```
従来: 形状の厳密性を判定基準
改善: 「ユーザーが何を表現したかったか」を大らかに推測
```

#### 3. **判断基準の変更**
```
従来: 判断に迷ったら候補を除外（conservative）
改善: 判断に迷ったら「必ず残す」（inclusive）
```

#### 4. **候補分析の構造化**
```
従来: "has_features": true/false
改善: "status": "keep" / "drop"
     + "reasoning": 許容ルールに基づいた理由
```
# VLM ドキュメント統合（PROMPT_IMPROVEMENT）

最終更新: 2026年5月19日

目的は以下のとおりです。

## 1. 概要

スケッチ（ユーザー入力）と SBIR が返す上位候補（Top5）を入力として、VLM（Gemini）により「特徴抽出→候補フィルタリング→アクション決定（execute/ask_user）」を行います。
本ドキュメントはプロンプト設計方針、出力形式、セットアップ手順、運用上の注意点をまとめています。

## 2. プロンプト設計方針（抽象化）

目的: どんな物体カテゴリにも適用できる「カテゴリ非依存」の特徴抽出を行い、VLMが偏らない判断を行う。

- 明示的な物体名・色名はプロンプトに含めない。
- スケッチの「構造的特徴」「シルエット」「トポロジー」「表面特性」「相対スケール」を抽象的に定義する。
- ユーザーの描写が不正確である前提で、判断に迷う箇所は除外せず候補を残す（inclusive）。
- 構造的に決定的な特徴（例: ある種の突出や穴の有無）が明確に示されている場合はフィルタリングに反映するが、描画誤差による詳細は除外の理由にしない。

### 推奨する抽出特徴（カテゴリに依存しない例）
- 基本的シルエット（輪郭、プロポーション）
- 構成要素の有無と配置（主要パーツの相対関係）
- トポロジー（連結性、穴の有無など）
- 表面特性（質感の印象、反射性、繰り返しパターンの有無）
- 相対的スケール感（候補間でのサイズ差の比較）

## 3. 出力 JSON 仕様（必須）

VLM は常に有効な JSON のみを返すこと。出力例（抽象化）:

```json
{
  "extracted_features": "カテゴリ非依存の特徴記述",
  "candidate_analysis": [
    {"id": 1, "status": "keep" | "drop", "reasoning": "特徴量との合致理由"},
    {"id": 2, "status": "keep" | "drop", "reasoning": "..."},
    {"id": 3, "status": "keep" | "drop", "reasoning": "..."},
    {"id": 4, "status": "keep" | "drop", "reasoning": "..."},
    {"id": 5, "status": "keep" | "drop", "reasoning": "..."}
  ],
  "remaining_candidates": [1,3],
  "action_type": "execute" | "ask_user",
  "selected_id": null | 1,
  "reasoning": "最終判断の理由（特徴量ベース）",
  "robot_question": null | "ユーザーへの逆質問（構造・形状・表面性についてのみ）"
}
```

注意:
- `robot_question` は具体的な物体名や色名を含めないこと（例: 「取っ手はありますか？」「平らな面が見えますか？」のように構造や形状に焦点を当てる）。
- `selected_id` は UI 側表示に合わせて 1..5 の形式で返す。

## 4. セットアップ（要点）

1. Gemini API キーを取得し、環境変数 `GEMINI_API_KEY` に設定する。
2. 必要パッケージをインストール: `pip install google-genai`
3. オプション: `ENABLE_VLM=true/false` で VLM 処理を制御する。

## 5. 統合フロー（概要）

ユーザースケッチ → `sub_writing1.py`（SBIR 実行）→ Top5 を `gemini_api.py` に渡す → VLM が JSON を返す → `touch.html` 等のフロントエンドへ配信。

## 6. 運用上のルール（重要）

- 明確に欠落している主要構成要素のみ `drop` 可とする。
- 構成要素が共通している候補の微細差（線の閉じ／開き、接合の微小差）は `drop` 理由にしない。
- フィルタリング結果で `keep` が複数ある場合、VLM は独断で `execute` にすべきではない（`ask_user` を推奨）。

## 7. トラブルシューティング（要点）

- `GEMINI_API_KEY` 未設定: 環境変数を確認
- JSON パースエラー: VLM の応答がコードブロック等でラップされていないか確認。必要に応じてコード側で ```json ブロックを取り除く処理を入れる。
- レイテンシ: 画像解像度を下げる、`ENABLE_VLM=false` にするなどで緩和
